# Phase 5 — UI Design Contract: Dashboard — Member Overview

> Visual and interaction contract for the admin dashboard member list, risk counters, and check-in history. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Layout & Interaction Contracts

### Dashboard Overview (Member List + Counters)

**Page structure:**
1. **Header section** (bg-gray-50, p-8)
   - Page title: "Membros"
   - Three counter badges (horizontal flex)
   - Primary CTA: "+ Novo Membro"

2. **Counter badges** (each)
   - Number (large, font-semibold)
   - Label (small, text-gray-600)
   - Optional: brief duration subtext

3. **Member table** (bg-white, border border-gray-200, rounded-lg)
   - Columns: Name, Email, Days away, Risk level
   - Rows sorted by days-away DESC (longest absent first)
   - Row hover: bg-gray-50, cursor-pointer
   - Name is a clickable link to member detail page

4. **Empty state** (bg-white, border, rounded, p-12, text-center)
   - Heading + body text + link to create first member

**Data sorting:**
- Primary: `days_since_last_checkin DESC` (longest absent first)
- Never-checked-in members appear at top (nullsFirst behavior)
- Active members only (filtered by status = 'active')

**Risk level calculation** (in component or query):
- Active: days ≤ 4
- At-risk: 4 < days ≤ 7
- Inactive: days > 7

**Page structure:**
1. **Breadcrumb navigation** (bg-gray-50, p-8)
   - Membros > [Member Name]

   - Basic info grid, status badge, action buttons
   - (No changes from existing member profile page)

**Data loading:**

## Data Contracts

### Member Risk Calculation (Server-side)

```typescript
// Pseudocode for dashboard query
const members = await supabase
  .from('members')
  .select('id, name, email, last_checked_in, status')
  .eq('org_id', org_id)
  .eq('status', 'active')
  .order('last_checked_in', { ascending: true, nullsFirst: true })

// Executor calculates days and risk level:
// - days_since = NOW() - last_checked_in
// - risk_level: 'active' | 'at_risk' | 'inactive'
// - Counters: sum by risk_level
```

### Check-In History (Server-side)

```typescript
// Pseudocode for history query
const page = 1
const pageSize = 50
const checkins = await supabase
  .from('checkins')
  .select('id, checked_in_at, ip_address, user_agent', { count: 'exact' })
  .eq('member_id', memberId)
  .order('checked_in_at', { ascending: false })
  .range((page - 1) * pageSize, page * pageSize - 1)

// Response includes:
// - checkins: Checkin[]
// - count: total number of records
// - totalPages = ceil(count / 50)
```

## State Management & Server Components

**Per requirements:** All data fetching via Server Components (no useEffect).

**Dashboard page flow:**
1. Server Component fetches org_id from JWT
2. Server Component queries members, sorts by days, calculates risk
3. Renders static HTML + counters
4. No JavaScript required for initial load

**Member detail page flow:**
1. Server Component fetches member profile (existing)
2. Server Component fetches first page of check-ins (new)
3. Pagination: either
   - Option A: Server-side (link to ?page=2, new query)
   - Option B: Client-side (fetch all in initial query if count < 500)

Executor chooses pagination approach based on performance.
