> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | Tailwind CSS (no design system library) |
| Preset | not applicable |
| Component library | none (vanilla HTML + Tailwind utilities) |
| Icon library | Not specified for Phase 2 |
| Font | Inter (Google Fonts, already loaded in root layout) |

**Decision source:** CONTEXT.md D-04, D-05, D-06 — Minimal centered card layout, no component library required for Phase 2.

**Rationale:** Phase 2 is authentication only (signup, login, password reset, logout). These are simple forms that don't require a full component library. Future phases (3+) may introduce a shadcn preset if more complex UI emerges. Starting with vanilla Tailwind keeps the build minimal and allows flexibility later.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing, input padding (px-4 py-2 = 16px) |
| lg | 24px | Section padding (p-6 in Tailwind = 24px), form field margins |
| xl | 32px | Card horizontal padding, layout gaps |
| 2xl | 48px | Major section breaks, page-level gutters |
| 3xl | 64px | Page-level spacing |

Exceptions: None. All spacing uses the 8-point scale.

**Decision source:** Tailwind default spacing scale. Standard 8-point grid across all future phases.

## Accessibility & Semantic HTML

**Form inputs:**
- Each input must have a corresponding `<label>` element with `for` attribute
- Email inputs use `type="email"` for native validation and keyboard affordances
- Password inputs use `type="password"`
- All inputs have `required` attribute

**Error messages:**
- Displayed in `<p>` tags with `class="text-red-500 text-sm"`
- Placed immediately below the related input for screen reader context
- Alternative: Use `aria-describedby` to link error message to input if using ARIA

**Buttons:**
- Always have text content (no icon-only buttons in Phase 2)
- Disabled state uses `disabled` attribute
- Loading state shows pending text in button label

**Links:**
- Use semantic `<a>` tags with `href` attribute
- No JavaScript-only navigation (use `<a>` not `<button>` for navigation)

**Color not the only cue:**
- Error states combine color + text message
- Required fields marked with text ("required") or visual indicator, not color alone

**Decision source:** Standard WCAG 2.1 AA guidelines. No explicit accessibility requirements in CONTEXT.md, but following best practices for form accessibility.

## Notes for Executor

**Key implementation points (read before starting):**

1. **Auth route group:** Create `src/app/(auth)/layout.tsx` to wrap all auth pages in the centered card container. Root layout doesn't change.

2. **Form handling:** Use `useActionState` hook (React 19) to manage server action state and loading pending flag. This avoids react-hook-form dependency.

3. **Error display:** Errors go inline below the input field in red-500. Check RESEARCH.md §Pattern 5 for exact implementation.

4. **Org creation:** On signup, immediately create an `organizations` record and set `app_metadata.org_id` via service role client. This is critical for RLS in future phases.

5. **Middleware matcher:** Protect both `/dashboard/:path*` AND `/api/admin/:path*` routes. Use `getUser()`, not `getSession()`.

6. **Password reset:** Multi-step flow: (1) request email with `resetPasswordForEmail()` redirecting to `/auth/reset-password?token_hash=X&type=recovery`, (2) verify token with `verifyOtp()`, (3) update password with `updateUser()`.

7. **Sidebar layout:** Build `DashboardLayout` reusable across Phase 3+ without modification. Test that future phases inherit the layout correctly.

8. **Tailwind class names:** All spacing uses `px-4`, `py-2`, `p-8`, `space-y-4` (NOT custom rem values). All colors use Tailwind names: `emerald-600`, `gray-50`, `red-500`, etc. No custom CSS.

9. **Portuguese language:** Every visible string must be in pt-BR. Check REQUIREMENTS.md for language context.

10. **No external components:** Phase 2 uses vanilla HTML + Tailwind only. No imports from component libraries.
