### CheckinForm Client Component

**Props from Server Component:**
```typescript
{
  orgId: string;
  orgName: string;
  qrHash: string;
}
```

**Form structure:**
1. CPF Input field
   - Label: (none visible; implicit in placeholder)
   - Input placeholder: "000.000.000-00"
   - Input height: 40px (`h-10`)
   - Input padding: 8px 12px (sm + xs)
   - Input border: `border border-gray-300`
   - Input border radius: `rounded-md`
   - Input font: 14px normal
   - Focus state: `focus:ring-2 focus:ring-emerald-600 focus:border-transparent`
   - Mask: Formatted input mask displaying as `___.___.___-__` (client-side JS; see Claude's Discretion)
   - Max length: 14 characters (accommodates formatted input)

2. Error message (below input)
   - Display: Inline, red-500 (`text-red-500`), 12px
   - Margin: 4px top

3. Submit button ("Confirmar Check-in")
   - Style: emerald-600 background, white text
   - Size: full-width, height 40px (`h-10`)
   - Padding: 8px 16px (sm + md)
   - Border radius: `rounded-md`
   - Font: 14px semibold
   - Hover: `hover:bg-emerald-700`
   - Active/loading state: disabled, opacity reduced, button text unchanged (no spinner; simple disable)
   - Margin: 16px top

**Interaction:**
- User types CPF; input masked client-side
- User taps "Confirmar Check-in"
- Button disabled during fetch
- On success → replaced with Success Screen (see below)
- On error (DUPLICATE, NOT_FOUND, etc.) → error message shown below input
- User can correct and retry, or close tab

## Form Validation Contract

### Client-side Validation (JavaScript)

- **CPF format:** Required, exactly 11 digits (after stripping non-digits)
- **Empty input:** Show "CPF é obrigatório"
- **Invalid length:** Show "CPF deve ter 11 dígitos"
- **Checksum validation:** Deferred to Phase 9

### Server-side Validation (POST /api/checkin)

- **QR hash:** Lookup in `organizations.qr_code_hash` — must exist
- **CPF:** Lookup in `members.cpf` with matching `org_id` — must exist
- **Member status:** Must be `status = 'active'` — inactive members rejected
- **Duplicate check:** Query `checkins` for `member_id` and `checked_in_at` within last 4 hours — reject if found
- **Org ID:** Derived from QR hash lookup (never trust client-provided org_id)

**Response codes:**
- `200 { ok: true, memberName: string, checkedInAt: ISO string }` — success
- `400 { ok: false, code: 'NOT_FOUND' }` — CPF not found or member inactive
- `400 { ok: false, code: 'DUPLICATE', checkedInAt: ISO string }` — check-in within 4h
- `400 { ok: false, code: 'INVALID_HASH' }` — QR hash invalid (client-side validation should catch first)
- `500` — server error (should not happen; logged on server)

## Audit Trail & Accessibility

### Audit Trail

**Recorded per check-in:**
- `member_id` — foreign key to members table
- `org_id` — from organization (QR hash lookup)
- `checked_in_at` — current UTC timestamp
- `ip_address` — from `x-forwarded-for` header
- `user_agent` — from `User-Agent` request header

### Accessibility

**Keyboard Navigation:**
- Tab order: CPF input → button
- Button responds to Enter and Space

**ARIA Labels:**
- Form input has `aria-label="CPF"` or `aria-labelledby` to implicit label
- Error messages associated via `aria-describedby`
- Success heading announced to screen readers

**Focus States:**
- CPF input: emerald-600 focus ring (2px)
- Button: underline or color change on focus

**Color Contrast:**
- emerald-600 text on white card: 5.7:1 (WCAG AA ✓)
- red-500 error text on white: 5.9:1 (WCAG AA ✓)
- gray-600 secondary text on white: 5.2:1 (WCAG AA ✓)
- White text on emerald-600 button: 7.8:1 (WCAG AAA ✓)

## API Integration

### POST /api/checkin Route Handler

**Request:**
```typescript
{
  qr_hash: string;
  cpf: string; // 11 raw digits (non-formatted)
}
```

**Response (success):**
```typescript
{
  ok: true;
  memberName: string; // Full name for welcome message
  checkedInAt: string; // ISO 8601 timestamp for display
}
```

**Response (failure):**
```typescript
{
  ok: false;
  code: 'NOT_FOUND' | 'DUPLICATE' | 'INVALID_HASH';
  checkedInAt?: string; // Provided only for DUPLICATE (previous check-in time)
}
```

**Headers used:**
- `x-forwarded-for` — IP address (set by Vercel)
- `User-Agent` — Browser/device identifier
- `Content-Type: application/json`

## Server Component Pattern

The check-in page (`/checkin/[hash]/page.tsx`) is a **Server Component** that:

1. Validates QR hash upfront against `organizations.qr_code_hash`
2. If invalid → renders error card inline
3. If valid → passes `{ orgId, orgName, qrHash }` to Client Component `<CheckinForm />`
4. Never renders a redirect; always renders an inline error card

This pattern matches Phase 3's QR code page split (`page.tsx` + `QRCodeDisplay.tsx`).
